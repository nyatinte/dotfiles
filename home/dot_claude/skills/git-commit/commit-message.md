# commit メッセージガイドライン

このドキュメントは、 commit メッセージ作成時のガイドラインを記載します。

## 基本原則

### Whyを重視

commit メッセージでは**Why**（なぜこの変更が必要か）に焦点を当てる。（どんな変更をしたかは commit の diff でわかるので、それを含める必要はない）

**Good**:

```md
feat: カウンセリングシート設定画面を追加

ユーザーからの要望により、カウンセリングシートを
カスタマイズできる設定画面が必要になったため実装。
```

**Bad**:

```md
feat: 画面追加

CounselingSettingPage.tsxを追加しました。
```

WHY が不明瞭な場合は、いくつか推測し、 AskUserQuestion Tool でユーザーに選択してもらう

### 簡潔だが説明的

- 1行目: 要約（70文字以内推奨）
- 2行目以降: 詳細（必要に応じて）
- 書き過ぎはよくない

### ユーザーの過去の Commit メッセージの分析

ユーザーの過去の Commit メッセージを分析して、プロジェクトのスタイルに合わせる:

```sh
git log --oneline -10 --author="$(git config user.email)"
```

## commit メッセージフォーマット

### Conventional Commits形式

このプロジェクトでは以下の接頭辞を使用:

- `feat:` 新機能追加
- `fix:` バグ修正
- `refactor:` リファクタリング（機能変更なし）
- `style:` コードスタイル変更（フォーマット、セミコロンなど）
- `test:` テスト追加・修正
- `docs:` ドキュメント変更
- `chore:` ビルドプロセス、補助ツールの変更
- `perf:` パフォーマンス改善
- `ci:` CI/CDの改善

### 例

#### 機能追加

```md
feat: ユーザー認証機能を追加

JWTベースの認証機能を実装。
セキュリティ要件により、セッションベースから移行。
```

#### バグ修正

```md
fix: ログイン時のエラーハンドリングを修正

nullチェック漏れにより500エラーが発生していた問題を修正。
```

#### リファクタリング

```md
refactor: useErrorフックを共通化

複数コンポーネントで重複していたエラーハンドリングを
共通フックに集約。メンテナンス性向上のため。
```

#### テスト追加

```md
test: CounselingSettingPageのテストを追加

ByRole APIを使用したアクセシビリティテストを実装。
```

## commit メッセージ草案生成のポイント

### 1. 変更内容の分類

変更されたファイルから、変更の性質を判断:

- **新規ファイル追加**: `feat:`
- **既存ファイル修正**:
  - バグ修正なら `fix:`
  - リファクタリングなら `refactor:`
  - パフォーマンス改善なら `perf:`
- **テストファイル**: `test:`
- **ドキュメント**: `docs:`

### 2. 変更の背景を推測

以下の観点から背景を推測:

- **パフォーマンス改善**: ループ最適化、キャッシュ追加
- **互換性維持**: 古いAPIの残存、フォールバック処理
- **クリーンアップ**: 未使用コードの削除
- **セキュリティ**: 脆弱性対応、バリデーション追加
- **ユーザー要望**: 新機能追加、UI改善

### 3. 複数の草案を作成

ユーザーが選びやすいように、以下のバリエーションを用意:

1. **簡潔版**: 1行で要約
2. **詳細版**: 背景情報を含む複数行
3. **技術的版**: 技術的な詳細を含む

**例**:

変更: `CounselingSettingPage.tsx`を追加

**草案1（簡潔版）**:

```md
feat: カウンセリングシート設定画面を追加
```

**草案2（詳細版）**:

```md
feat: カウンセリングシート設定画面を追加

ユーザーからの要望により、カウンセリングシートを
カスタマイズできる設定画面を実装。
```

**草案3（技術的版）**:

```md
feat: カウンセリングシート設定画面を追加

GraphQL Mutation（updateCounselingSettings）と
連携したReactコンポーネントを実装。
フォームバリデーションにはZodを使用。
```

## 選択的ステージング

### 原則

タスクに直接関連するファイルのみをステージング。

### 手順

1. **変更ファイルを確認**: `git status`
2. **関連ファイルを特定**:
   - タスクに必要なファイル
   - 依存関係のあるファイル
3. **個別にステージング**: `git add <ファイル名>`
4. **ステージング内容を確認**: `git diff --cached`

### 避けるべき操作

- `git add .` - すべてのファイルをステージング（関連のないファイルが含まれる可能性）
- `git add -A` - 同上

### 例外

- 明らかにすべてのファイルが関連している場合のみ、`git add .`を使用可能

## pre-commitフック失敗時の対応

### フックが失敗した場合

1. **エラー内容を確認**:
   - リントエラー
   - フォーマットエラー
   - テスト失敗

2. **AskUserQuestionで確認**:

   ```md
   質問: 「pre-commitフックが失敗しました。どのように対応しますか?」
   選択肢:

   1. AIに修正を依頼する
   2. commit を再試行する
   3. --no-verifyを使用して commit を再試行する
   4. ユーザー自身が修正する
   ```

3. **選択肢に応じた対応**:
   - **1. AIに修正を依頼**: エラーを解析し、修正案を提示・実施
   - **2. commit を再試行**: 再度 `git commit` を実行
   - **3. --no-verifyを使用**: `git commit --no-verify` を実行（非推奨）
   - **4. ユーザー自身が修正**: 修正方法を案内して終了

### 重要な注意事項

- **--amendは使用しない**: フック失敗時、 commit は作成されていないため、--amendは前回の commit を修正してしまう
- **新規 commit を作成**: 修正後は新しい commit を作成

## commit メッセージのベストプラクティス

### Good

- Whyを明確にする
- 簡潔に書く（1行目は70文字以内）
- 能動態を使う（「追加した」ではなく「追加」）
- チケット番号を含める（該当する場合）

### Bad

- Whatだけを書く（「ファイルを変更」）
- 曖昧な表現（修正,更新,fix など）
- 不必要に長い説明
- 複数の無関係な変更を1つの commit にまとめる
