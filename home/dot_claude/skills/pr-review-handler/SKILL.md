# PR Review Handler

## 概要

このスキルは、GitHubのプルリクエストに寄せられたレビューコメントを体系的に取得し、各コメントに対して適切に対応するためのワークフローを提供します。`getreview.sh`スクリプトを使用してレビューコメントを取得し、問題を分析して修正案を提案し、コメント単位でコミットを作成し、手動での返信とresolveをガイドします。

## 使用タイミング

このスキルを使用する場合:

- PRにレビューコメントが寄せられ、体系的に対応したい場合
- 未解決のレビューコメントを整理して順番に対応したい場合
- 各コメントに対して修正とコミットを分けて管理したい場合
- レビューアーへの返信とresolveを適切に行いたい場合

## ワークフロー

### Step 1: レビューコメントの取得

以下の2つのスクリプトが利用可能です。状況に応じて使い分けてください:

#### `get-review-threads.sh`（推奨）
GraphQL APIを使用し、スレッドの解決/未解決ステータスを取得できます:

```bash
bash .claude/skills/pr-review-handler/scripts/get-review-threads.sh
```

このスクリプトは以下を実行します:

1. 現在のブランチに対応するPR番号を取得 (`gh pr view --json number`)
2. GitHub APIを使用してPRのレビューコメントを取得
3. スレッドごとにコメントをグループ化
4. ファイルパス、行番号、コメント内容を整形して表示

**出力形式:**

```text
📁 src/utils/helper.ts:42 (position: 123)
🧵 スレッドID: 123456
  🟢 reviewer_name: この関数はエラーハンドリングが不足しています。
  └─ 💬 author_name: 確認しました、修正します。
```

### Step 2: コメントの分析と分類

取得したコメントを分析し、以下の観点で分類します:

- **修正が必要なコメント**: コードの変更を伴う指摘
- **質問・確認コメント**: 説明や議論が必要なコメント
- **承認待ちコメント**: 既に対応済みで返信のみ必要なコメント
- **スキップ対象**: 後で対応するコメント

各コメントについて:

- ファイルパスと行番号を確認
- 該当コードを`Read`ツールで読み取り
- レビューアーの意図を理解
- 対応の優先度を判断

### Step 3: 修正案の生成

修正が必要なコメントに対しては、以下を実施します:

1. **コードの分析**
   - 指摘された箇所のコードを読み取り
   - 問題点を特定
   - 影響範囲を確認

2. **修正案の提案**
   - 具体的な修正内容を提示
   - 修正後のコード例を示す
   - 変更の理由を説明

3. **ユーザーへの確認**
   - 修正案を提示し、承認を得る
   - 必要に応じて代替案も提示

### Step 4: 修正の実装とコミット

修正案が承認されたら:

1. **コードの修正**
   - 指摘されたファイルを修正
   - 関連するファイルも必要に応じて修正
   - テストコードも更新（必要に応じて）

2. **コミットの作成**
   - コメント単位でコミットを作成
   - コミットメッセージは以下の形式を推奨:

     ```text
     fix: [ファイル名] [修正内容の要約]

     - [具体的な変更内容1]
     - [具体的な変更内容2]

     Refs: #PR番号
     ```

   - コミットメッセージの書式ルールに従う（git-commit Skill の `commit-message.md` を参照）

3. **変更の確認**
   - Lintエラーがないか確認（必要に応じて `pnpm lint` を実行）
   - 修正内容が意図通りかを確認

### Step 5: 返信とResolve

修正が完了したら:

1. **返信文の生成**
   - 修正内容を説明する返信文を作成
   - 質問への回答を生成
   - 必要に応じて議論を継続

2. **返信の投稿**
   - ユーザーに返信文を提示
   - 承認後、`gh api`で返信を投稿:

     ```bash
     gh api \
       repos/{owner}/{repo}/pulls/{pr_number}/comments/{comment_id}/replies \
       -f body="返信文"
     ```

3. **Resolveのガイド**
   - 修正が完了したコメントはresolveするよう案内
   - GitHub UIでのresolve手順を説明

### Step 6: 対応履歴の記録

各コメントへの対応内容を`REVIEW_NOTES_{PR_NUMBER}.md`に記録:

```markdown
# PR #{PR_NUMBER} レビュー対応履歴

生成日時: YYYY-MM-DD HH:MM

## 対応サマリー

- 総コメント数: N
- 修正完了: N
- 返信完了: N
- スキップ: N

---

## [1/N] src/utils/helper.ts:42

**レビュアー:** @reviewer_name
**対応:** 修正完了

**指摘内容:**

> この関数はエラーハンドリングが不足しています。

**修正内容:**

- nullチェックを追加
- エラーケースの処理を追加
- テストケースを追加

**コミット:** abc1234 - fix: helper.tsのエラーハンドリングを追加

**返信:**

> ご指摘ありがとうございます。nullチェックとエラーケースの処理を追加しました。

---
```

## 重要な原則

1. **体系的に対応**: コメントを順番に処理し、漏れなく対応する
2. **コメント単位でコミット**: 1つのコメントに対する修正は1つのコミットにまとめる
3. **明確なコミットメッセージ**: 何を修正したかが分かるメッセージを書く
4. **適切な返信**: 修正内容を説明し、レビューアーに感謝を示す
5. **履歴の記録**: 全ての対応内容を記録し、後で参照できるようにする

## ベストプラクティス

- **優先度の判断**: 重要な指摘から先に対応する
- **関連コメントの統合**: 同じファイルの関連するコメントはまとめて対応する
- **テストの更新**: 修正に伴いテストも更新する
- **コミットの粒度**: 1つのコメント = 1つのコミットを基本とする
- **返信のタイミング**: 修正が完了してから返信する

## 注意事項

- PR番号の取得は現在のブランチから自動的に行われる
- スレッドごとにグループ化されるため、返信コメントも含めて表示される
- 未解決のコメントのみを対象とする場合は、`get-review-threads.sh`（GraphQL版）を使用して`isResolved`ステータスでフィルタリングする
