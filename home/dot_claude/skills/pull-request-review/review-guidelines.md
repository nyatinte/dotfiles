# レビューコメントガイドライン

## レビュー出力フォーマット

### 全体構成

```markdown
# PR #{number} レビュー

## 概要

PRの変更内容を1-2文で要約する。

---

## {バッジ} {タイトル}

**ファイル**: `path/to/file.tsx:line_number`

指摘内容。

---

...(続き)
```

### バッジ

![badge-IMO](https://img.shields.io/badge/review-IMO-orange) のようなバッジを使用してください。

## レビューの観点

コーディングガイドラインおよびPRで明記された仕様に基づき、客観的視点でレビューを行ってください。

### 確認しない項目

- 自動生成ファイル（**generated**）の内容
- フォーマット（Prettierが管理）
- PRの変更に関係ない既存コードの問題

## 表現のガイドライン

### 協調的な表現を使う

**避ける表現:**

- "これは間違いです"
- "こうすべきです"
- "修正してください"

**使う表現:**

- "〜と思いました！"
- "〜があると良さそうです"
- "〜の方が良いかもしれません"
- "〜かな…と思ったりもしました"

### 実装コストへの配慮

指摘が大きな変更を伴う場合は配慮を示す:

- "影響範囲もありそうなので無理せずで問題ないです！"
- "余裕があれば〜"
- "今回のスコープ外かもですが、将来的に〜"

### 簡潔さを保つ

- 1つの指摘は短く（3-5文以内）
- コード例は必要な場合のみ
- 複数の代替案を詳細に説明しない（最も良いと思う1つを提案する）

## アンチパターン

- 代替案A/B/Cを詳細に比較する長文コメント
- 深いネスト構造の見出し（h2まで）
- 断定的な表現
- 自明な点の過剰説明
- 変更に関係ない既存コードへの指摘
