# Strategic Compact スキル

自動 compaction に任せるのではなく、ワークフローの適切なタイミングで手動 `/compact` を提案します。

## いつ使うか

- セッションが長くなりコンテキスト上限 (200K+ トークン) に近づいているとき
- フェーズをまたぐ作業中 (調査 → 計画 → 実装 → テスト)
- 同一セッション内で無関係なタスクに切り替えるとき
- 主要マイルストーン完了後に次の作業を始めるとき
- レスポンスが遅くなったり精度が落ちてきたとき (コンテキスト圧迫)

## なぜ戦略的 compaction か

自動 compaction は任意の地点で発生します:

- タスクの途中で発生し、重要なコンテキストが失われることがある
- タスクの論理的な境界を認識しない
- 複雑なマルチステップ操作を中断してしまう可能性がある

論理的な境界での戦略的 compaction:

- **調査後・実装前** — 調査コンテキストを圧縮し、実装計画を残す
- **マイルストーン完了後** — 次のフェーズをフレッシュな状態で開始
- **大きなコンテキスト切り替え前** — 調査コンテキストをクリアしてから別タスクへ

## 仕組み

`suggest-compact.js` は PreToolUse (Edit/Write) で実行され:

1. **ツール呼び出し回数を追跡** — セッション内の呼び出し数をカウント
2. **閾値検知** — 設定可能な閾値 (デフォルト: 50回) で compaction を提案
3. **定期リマインド** — 閾値超過後は 25 回ごとにリマインド

## 環境変数

- `COMPACT_THRESHOLD` — 最初の提案までのツール呼び出し回数 (デフォルト: 50)

## compaction 判断ガイド

| フェーズ遷移 | compaction? | 理由 |
|---|---|---|
| 調査 → 計画 | Yes | 調査コンテキストは重い。計画が凝縮された出力 |
| 計画 → 実装 | Yes | 計画は TodoWrite やファイルにある。コード用に空きを作る |
| 実装 → テスト | 場合による | 最近のコードを参照するなら維持。切り替えなら compact |
| デバッグ → 次の機能 | Yes | デバッグのトレースは無関係な作業の邪魔になる |
| 実装の途中 | No | 変数名・ファイルパス・部分的な状態を失うコストが高い |
| 失敗したアプローチの後 | Yes | 行き詰まりの推論をクリアしてから新アプローチへ |

## compaction 後に残るもの

何が残るかを把握することで、安心して compact できます:

| 残るもの | 失われるもの |
|---|---|
| CLAUDE.md の指示 | 途中の推論と分析 |
| TodoWrite のタスクリスト | 以前に読んだファイルの内容 |
| メモリファイル (`~/.claude/memory/`) | マルチステップの会話コンテキスト |
| Git の状態 (コミット・ブランチ) | ツール呼び出し履歴とカウント |
| ディスク上のファイル | 口頭で伝えたユーザーの細かい好み |

## ベストプラクティス

1. **計画後に compact** — TodoWrite に計画が確定したらフレッシュスタート
2. **デバッグ後に compact** — エラー解決のコンテキストをクリアしてから続行
3. **実装途中は compact しない** — 関連する変更のコンテキストを維持する
4. **提案を読む** — フックは「いつか」を教えてくれる。「するかどうか」はあなたが決める
5. **compact 前に書き出す** — 重要なコンテキストはファイルやメモリに保存してから
6. **サマリー付きで `/compact`** — 例: `/compact 次は認証ミドルウェアの実装に集中`
